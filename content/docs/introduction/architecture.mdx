## System Architecture

### Application Structure

The platform follows a modern serverless architecture with clear separation of concerns.

```
┌─────────────────────────────────────────────┐
│           Client Browser                     │
│  (React Components + Next.js Pages)         │
└──────────────┬──────────────────────────────┘
               │
               │ HTTPS
               ▼
┌─────────────────────────────────────────────┐
│         Next.js Server                       │
│  - Server Components                         │
│  - API Routes                                │
│  - Server Actions                            │
└──────────────┬──────────────────────────────┘
               │
       ┌───────┴────────┐
       │                │
       ▼                ▼
┌──────────┐    ┌──────────────┐
│  Clerk   │    │   MongoDB    │
│  Auth    │    │   Database   │
└──────────┘    └──────────────┘
```

### Data Flow

<Callout title="Request Flow" type="success">
User interactions flow through client-side validation, server actions for business logic, Prisma ORM for database access, and back to updated UI state—ensuring type safety at every step.
</Callout>

**User Action Workflow**

1. **User Interaction**: Component event handler triggered
2. **Client Validation**: Form validation and immediate UI feedback
3. **Server Action**: Next.js server action processes request
4. **Business Logic**: Server-side validation and processing
5. **Database Query**: Prisma ORM queries MongoDB
6. **Response**: Updated UI state with new data

**Real-Time Features**

- **Chat System**: Server-sent events enable real-time messaging
- **Notifications**: Automatic UI updates on data changes
- **Status Updates**: Live application status tracking
- **Presence Indicators**: Online/offline user status

## Data Management

### Database Design Principles

<Callout title="Schema Design" type="info">
Our database schema balances normalization for data integrity with strategic denormalization for read performance, ensuring optimal query efficiency.
</Callout>

**Design Approach**

- **Denormalization**: Optimized for read-heavy workloads
- **Embedded Documents**: Related data stored together for atomic operations
- **References**: Relationships maintained where data independence is required
- **Indexing Strategy**: Optimized indexes for common query patterns

### Core Data Models

**User Model**

```typescript
{
  id: string
  clerkId: string              // Clerk authentication ID
  email: string
  role: 'STUDENT' | 'BUSINESS_OWNER' | 'ADMIN'
  profile: Profile             // One-to-one relation
  projects: Project[]          // For business owners
  applications: Application[]  // For students
  createdAt: DateTime
  updatedAt: DateTime
}
```

**Project Model**

```typescript
{
  id: string
  title: string
  description: string
  category: string
  skills: string[]
  status: 'OPEN' | 'IN_PROGRESS' | 'COMPLETED' | 'CLOSED'
  owner: User                  // Relation to business owner
  applications: Application[]
  createdAt: DateTime
  deadline: DateTime
}
```

**Application Model**

```typescript
{
  id: string
  student: User                // Relation to student
  project: Project             // Relation to project
  status: 'PENDING' | 'ACCEPTED' | 'REJECTED' | 'WITHDRAWN'
  coverLetter: string
  createdAt: DateTime
  updatedAt: DateTime
}
```

## API Architecture

### API Endpoint Structure

**Public Endpoints**

```
GET  /api/projects           # List all available projects
GET  /api/projects/[id]      # Get specific project details
```

**Protected Endpoints**

```
POST   /api/projects         # Create project (business only)
PATCH  /api/projects/[id]    # Update project (owner only)
DELETE /api/projects/[id]    # Delete project (owner only)

POST   /api/applications     # Submit application (student only)
PATCH  /api/applications/[id] # Update application status
```

**Webhook Endpoints**

```
POST /api/webhooks/clerk     # Clerk user synchronization
```
